## Context

The existing current-runtime has useful pieces: leases record creation and ACK
times, `progress` records metadata-only ongoing work, Controller has a timed
patrol, and tests already prove that progress can preserve a slow result wait.
The observed failure came from a separate host-liveness status path where
`timeout_unknown` became replacement evidence even though the active role had
recent current-run activity and no formal progress reminder loop had been
exhausted.

The repair should reduce surfaces, not add another compatibility layer. The
current wait authority becomes the packet/lease ledger plus result acceptance:
final result wins, ACK starts result wait, progress refreshes liveness evidence,
and evidence age selects wait/remind/replace.

## Goals

- Keep one uniform background-role policy for PM, worker, reviewer,
  FlowGuard operator, and other packet holders.
- Use only ACK and progress as current liveness evidence.
- Keep Controller mechanical: it asks runtime, then performs the runtime-owned
  wait, reminder, or replacement duty.
- Delete unsupported host-liveness/timeout fields from current contracts.
- Preserve sealed-body boundaries; progress remains metadata-only and never
  becomes completion or quality evidence.
- Add Cartesian fake-agent coverage for the new ladder and legacy-field
  rejection.

## Non-Goals

- No role-specific wait thresholds.
- No compatibility shim that translates `timeout_unknown` into a current
  replacement decision.
- No new durable liveness ledger when existing lease ACK/progress fields can
  express the evidence.
- No Controller-side log reading, chat-history inference, or sealed-body
  inspection to decide liveness.

## Runtime Policy

The runtime guard computes a single current liveness evidence timestamp:

```text
before ACK:
  evidence_at = lease.created_at
  ack status = missing

after ACK:
  evidence_at = max(lease.ack_received_at, lease.last_progress_at)
  evidence_kind = ack or progress
```

For `wait_for_ack`:

```text
elapsed since lease.created_at < 5 min      -> wait_patrol
5 min <= elapsed < 10 min                  -> remind_ack
elapsed >= 10 min                          -> reissue_or_replace_lease
```

For `wait_for_result`:

```text
accepted result exists                     -> result path handles completion
elapsed since evidence_at < 10 min         -> wait_patrol
10 min <= elapsed < 30 min                 -> remind_progress
elapsed >= 30 min                          -> reissue_or_replace_lease
```

The foreground duty exposes fixed reminder text generated by runtime:

- ACK reminder: ask the role to ACK the current packet or submit a blocker if it
  cannot receive it.
- Progress reminder: ask the role to submit the final result if complete,
  immediately write `progress +1` if still working, keep progress frequent when
  work continues, or submit a blocker if unable to continue.

## Field Disposition

Current wait replacement must not read these as valid authority:

- `timeout_unknown`
- `timeout_unknown_is_active`
- `timeout_unknown_is_not_active`
- `timeout_unknown_role_keys`
- `wait_agent_timeout_is_timeout_unknown_not_active`
- `bounded_wait_result=timeout_unknown`
- lease `liveness_status`
- lease `last_liveness_status`
- `host_liveness_status` for current role wait decisions
- `host_liveness_history`
- `host_liveness_reports`

The replacement fields are existing current-contract fields:

- `created_at`
- `ack_received`
- `ack_received_at`
- `progress_count`
- `last_progress_at`
- `last_progress_status`
- `accepted_result_id`

If runtime outputs `last_liveness_evidence_at` or
`last_liveness_evidence_kind`, those outputs are derived guard diagnostics and
not a second persisted authority.

## Prompt And Role Contract

Every role card and packet identity prompt must state the same behavior:

- ACK is receipt only.
- If final output is not ready, the role should write progress whenever work
  starts, resumes, reaches a small milestone, begins or finishes a long command,
  receives a runtime reminder, or is otherwise still working.
- Progress must be short metadata only and must not contain sealed content,
  findings, evidence, recommendations, decisions, or final result details.
- Final completion still requires the assigned result path.
- Inability to continue requires the current blocker path.

## Testing And Fake-Agent Matrix

The fake-agent matrix is generated over these dimensions:

- ACK state: no ACK, fresh ACK, stale ACK.
- Result state: no result, valid final result, invalid final result.
- Progress state: none, fresh, 10-30 minute stale, 30+ minute stale, wrong
  lease, wrong packet, after accepted result.
- Legacy pollution: none, `timeout_unknown`, host-liveness timeout,
  bounded-wait timeout, last-liveness-only.
- Time bucket: 0-5, 5-10, 10-30, 30+ minutes from the relevant evidence.
- Reminder history: none, ACK reminder, one progress reminder, repeated progress
  reminders.

Each generated case must assert runtime decision, Controller duty text,
legacy-field rejection, and positive recovery after a valid progress event.

## Validation Strategy

- Focused runtime lifecycle tests for boundary thresholds and old-field
  rejection.
- Prompt/card coverage tests proving every role receives the same progress
  instruction.
- Fake-agent Cartesian replay tests for the matrix above.
- Field lifecycle checks proving old liveness fields are deleted, forbidden, or
  negative-only.
- FlowGuard lifecycle/process checks, then meta/capability regressions through
  the repository background log contract.
- Install sync and local install audit before final local version confidence.
