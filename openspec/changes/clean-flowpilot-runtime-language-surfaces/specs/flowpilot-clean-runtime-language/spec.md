## ADDED Requirements

### Requirement: Active runtime vocabulary is clean and topology-neutral
FlowPilot SHALL describe runtime help through requested responsibilities and
host-supported role bindings rather than old runtime-role, sidecar role, fixed
crew, role-binding, unsupported historical, or FlowPilot runtime terminology in current-authority
surfaces.

#### Scenario: Runtime prompts use role-binding language
- **WHEN** a FlowPilot role, Controller, or bootloader receives active prompt
  content from `SKILL.md`, the runtime kit, or card prompts
- **THEN** the prompt describes additional help as runtime role assistance,
  requested responsibilities, role bindings, or role-binding recovery
- **AND** it does not present runtime role assistance, sidecar roles, fixed role binding size,
  diagnostic router repair, or old protocol unsupported historical as current behavior

#### Scenario: Templates generate clean current state
- **WHEN** repository templates generate or document current `.flowpilot`
  runtime state
- **THEN** those templates use role-binding ledgers, role-binding memory, and
  requested-responsibility coverage terminology
- **AND** they do not create current fields or instructions named for
  runtime role assistance, sidecar roles, requested role-binding coverage, or old unsupported historical fallback

### Requirement: Historical input rejection does not preserve current unsupported historical
FlowPilot SHALL reject unsupported historical channels or stale inputs without
advertising them as supported unsupported-run inspection, unsupported historical, or fallback paths.

#### Scenario: Unsupported channel is rejected neutrally
- **WHEN** a caller uses a unsupported command, stale route input, obsolete ACK
  channel, or old payload shape
- **THEN** the runtime or check reports an unsupported input/channel boundary
- **AND** the message does not instruct the user or role to continue through a
  unsupported historical, unsupported, or unsupported historical repair workflow

### Requirement: Current model obligations use requested role coverage
FlowPilot FlowGuard models SHALL represent role readiness as coverage of the
runtime-requested responsibilities for the current run, not as a fixed runtime-role
role binding or background-sidecar role count.

#### Scenario: Startup readiness is dynamic
- **WHEN** startup activation is modeled
- **THEN** readiness depends on requested role binding policy, required binding
  coverage, current-run binding evidence, and explicit fallback decisions
- **AND** it does not require `REQUIRED_ROLE_BINDING_COUNT = 6` or a role-binding policy as the
  modeled startup authority

#### Scenario: Resume recovery is binding recovery
- **WHEN** heartbeat or manual resume is modeled
- **THEN** recovery requires current-run role-binding memory and a
  role-binding recovery report before PM resume decisions
- **AND** it does not model recovery as full role binding recycle or role-binding
  rehydration

### Requirement: Clean-language validation is a release and install gate
FlowPilot SHALL provide validation that scans current prompt, template, runtime,
model, check, and install surfaces for old topology terms before local install
sync or completion is claimed.

#### Scenario: Old terminology fails the clean-language gate
- **WHEN** an active current-authority surface contains old runtime-role,
  sidecar role, fixed-crew, role-binding, unsupported historical alias, unsupported historical repair, or
  FlowPilot runtime terminology as current behavior
- **THEN** validation fails with the owning file and term category

#### Scenario: Historical archives are scoped
- **WHEN** a historical archive or result baseline contains old terminology
- **THEN** validation either excludes it as non-current evidence or records the
  owning current check that must regenerate it
- **AND** historical presence alone is not treated as active runtime authority
