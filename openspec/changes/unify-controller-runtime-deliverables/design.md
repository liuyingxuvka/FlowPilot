## Context

FlowPilot already has mature evidence primitives for formal role work:
system-card ACKs are read receipts only, formal packets carry output
contracts, role outputs are submitted through runtime-generated
envelopes/receipts, and Router validates path/hash evidence before accepting
events.

Controller stateful actions currently sit beside that path. A Controller action
row can declare `required_deliverables`, then Controller writes a local file and
a receipt. The live failure showed why this is fragile: the file existed, but
it was not the canonical shape Router expected, so Router could not reconcile
the action.

## Goals / Non-Goals

**Goals:**

- Reuse the existing runtime/output-contract evidence path for Controller
  formal deliverables.
- Keep Controller lightweight: the action ledger stays the work queue, and
  receipt-only actions stay receipt-only.
- Prevent Controller from hand-writing control-plane JSON that Router later
  treats as authoritative evidence.
- Keep Controller authority boundaries intact: Controller may create only
  allowed control-plane artifacts and may not read sealed bodies, approve gates,
  mutate routes, or act as a worker.
- Make repair accounting count failed returned repair evidence, not merely
  issued repair rows.

**Non-Goals:**

- Do not convert Controller into a PM/Worker role or require full work packets
  for every Controller action.
- Do not change the existing PM role-work process for reviewers, officers, or
  workers.
- Do not run heavyweight `run_meta_checks.py` or `run_capability_checks.py` for
  this focused repair.

## Decisions

1. **Controller action ledger remains the scheduler only**

   The ledger tells Controller which row is ready and where the output must go.
   It does not define a separate schema-validation system. Formal deliverable
   rows refer to an output contract/runtime artifact instead of open-ended
   `required_deliverables` JSON assembled by Controller.

   Alternative considered: make every Controller formal action a PM-style work
   packet. Rejected because Controller mechanical actions do not need sealed
   packet bodies, PM absorption, or reviewer review.

2. **Controller formal outputs use a narrow runtime profile**

   `role_output_runtime` gains Controller-only output types for mechanical
   control-plane artifacts. These output types are not general role reports:
   they generate or validate fixed artifacts, write a runtime receipt, and
   return envelope metadata that Router can check.

   Alternative considered: keep custom Router validators per Controller action.
   Rejected because it recreates the drift-prone mini-flow that failed.

3. **Router-owned mechanical artifacts are generated by runtime helpers**

   For `controller_boundary_confirmation`, the runtime/helper builds the
   canonical confirmation from Router-visible sources: manifest path/hash,
   controller core path/hash, policy hash, and boundary constraints. Controller
   submits the runtime envelope/receipt, not a hand-written JSON body.

   Alternative considered: let Controller copy a template. Rejected because the
   fields include hashes and policy values that should be recomputed, not
   transcribed.

4. **Router accepts runtime evidence, then applies state**

   Router reconciliation validates the runtime receipt and artifact hash, then
   syncs `controller_role_confirmed`,
   `controller_role_confirmed_from_router_core`, and
   `controller_boundary_confirmation_written`. A Controller receipt without
   valid runtime evidence remains incomplete.

5. **Repair attempts are counted on invalid returned evidence**

   Scheduling repair attempt N records a pending repair row but does not count
   as a failed attempt. The attempt becomes failed only when its receipt returns
   and runtime evidence is still missing or invalid.

## Risks / Trade-offs

- [Risk] Adding Controller to `role_output_runtime` could weaken role-output
  authority boundaries. -> Mitigation: allow only `controller` and only
  Controller-scoped output types with fixed control-plane contracts.
- [Risk] Runtime-generated artifacts can hide Controller action mistakes. ->
  Mitigation: Router still validates the runtime receipt, artifact hash, and
  expected action/contract before reconciling.
- [Risk] Existing `required_deliverables` fields are still present in older
  runs. -> Mitigation: support legacy repair detection, but make new Controller
  formal rows advertise runtime contract metadata and reject hand-written
  completion as incomplete.
